## 精简版（简历直接用）

**淘宝用户行为多维分析系统** | Hadoop + PySpark + Flask + ECharts

基于淘宝1亿条用户行为数据，使用 PySpark 构建10个分析模块（RFM分层、转化漏斗、行为路径等），通过 Flask 提供 REST API，前端 ECharts 渲染19个交互式图表。发现并解释了46.4%购买无浏览记录的数据集特征，将商品CVR从异常的705%修正为合理的2.25%。实现模块自注册架构，支持CLI选择性运行。

## 完整版（STAR 展开）

### Situation（背景）

电商平台上积累了海量用户行为数据（浏览、收藏、加购、购买），蕴含着用户消费偏好、商品热度趋势、转化漏斗特征等业务洞察。传统的单机分析工具面对亿级数据时存在处理速度慢、内存不足等问题，且缺乏系统化的多维分析框架。

数据集：阿里巴巴淘宝用户行为数据集，约1亿条记录，100万用户，9天时间窗口，原始CSV 3.67GB。

### Task（任务）

1. 构建完整的分布式数据处理流水线（清洗→分析→服务→可视化）
2. 实现10个维度的用户行为分析（用户分层、转化漏斗、行为路径、留存分析等）
3. 发现数据中的异常指标并给出合理解释
4. 设计可扩展的模块化架构，支持灵活添加/关闭分析模块

### Action（行动）

**数据清洗层**
- 使用 PySpark 定义 Schema 避免自动推断性能开销，过滤时间戳异常数据（约5万条），去重后以 Parquet 列式格式输出（3.67GB→1.9GB，压缩比2:1）

**分析计算层**
- 设计模块自注册机制：每个分析模块声明 MODULE_ID/MODULE_NAME/MODULE_ENABLED 元数据，通过 pkgutil 自动发现，实现三级启用控制（CLI参数 > config.yaml > 模块属性）
- 实现严格递进漏斗：通过 inner join 确保每阶段是上阶段的子集（pv→cart/fav→buy），解决传统独立统计导致的转化率虚高问题
- 实现 RFM 用户分层：R（最近购买时间）×0.4 + F（购买频率）×0.6 加权评分，5级分层，阈值和权重全部外部化至 config.yaml
- 实现会话级行为路径分析：引入30分钟会话分割阈值，使用 lag() 窗口函数按 (user_id, session_id) 分区，避免跨会话路径拼接

**服务接口层**
- Flask REST API 提供9个端点，支持日期范围过滤，启动时一次性加载 Parquet 数据至内存

**前端展示层**
- ECharts 19个交互式图表，ChartRegistry 注册表模式，Promise.all 并行加载数据，支持日期筛选、行为联动过滤、全屏查看

**异常排查**
- 发现67个商品CVR超过100%（最高705%），通过逐商品验证发现85-97%的购买者在数据期内无浏览记录，确认为购物车/收藏夹跨期效应而非计算错误
- 发现每日漏斗趋势 pv→interest 46%、interest→buy 42% 偏高，排查发现原代码三个指标独立统计而非递进，修复为按日期的递进漏斗后降至44%和24%

### Result（成果）

- 完成1亿条数据的分布式处理全流程，10个分析模块总耗时约30分钟（Spark local[2]模式）
- 输出22个JSON分析结果文件，19个ECharts交互式图表，覆盖用户价值、转化漏斗、行为路径等核心电商指标
- 发现并解释了数据集的系统性偏差：46.4%购买无当日浏览、商品CVR最高705%、整体CVR 68%（仅行为级CVR 2.25%可靠）
- 实现模块化架构：新增分析只需添加一个.py文件，关闭模块只需config.yaml写一行，CLI支持 --only/--skip/--enable 参数
- 前端从738行单文件重构为25个模块化文件，图表自注册、数据自动分发

## 面试高频追问 & 应答

**Q: 为什么用 PySpark 而不是 Hive/Flink？**
A: Hive 适合简单 SQL 聚合，但 RFM 分层、行为路径需要复杂窗口函数和多步 join，PySpark DataFrame API 更灵活。Flink 是流处理，本项目是离线批处理场景。

**Q: 漏斗为什么用 inner join 而不是独立统计？**
A: 独立统计的三个集合不保证包含关系，"pv→cart 转化率"实际上是"有 cart 的用户÷有 pv 的用户"，可能完全不重叠。inner join 确保每阶段是上阶段子集，转化率单调递减。

**Q: CVR 超过 100% 怎么解释？**
A: 不是计算错误，是数据集核心特征。85-97%的购买者在数据期内没有浏览记录——购物车/收藏夹是持久状态，用户在数据期前加购、数据期内直接下单。行为级CVR（buy笔数/pv笔数=2.25%）是唯一可靠指标。

**Q: 遇到过哪些性能问题？**
A: ①countDistinct 比 count 慢很多（需要额外 shuffle），task2 从15分钟增至40分钟；②HDFS 默认副本因子3导致3.6GB数据实际占用11GB，磁盘不足；③YARN 资源上限2048MB，executor 内存请求超限。分别通过增加超时、降低副本因子、回退 local 模式解决。

**Q: 如果数据量增长到10亿条怎么扩展？**
A: HDFS 副本恢复为3并增加节点；切回 YARN 模式增加 executor；增加 shuffle 分区数避免数据倾斜；引入增量更新机制避免全量重跑；如需实时化引入 Spark Streaming。

**Q: 模块自注册机制怎么设计的？**
A: 每个模块声明 MODULE_ID/MODULE_ENABLED 等元数据，__init__.py 通过 pkgutil.iter_modules 扫描、importlib.import_module 动态加载。三级启用控制：CLI > config.yaml disabled > MODULE_ENABLED。新增模块只需一个.py文件，无需修改主文件。
